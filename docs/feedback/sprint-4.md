## 実装アプローチ上の観察

- Planner の spec では「Claude API（embeddings または検索特化プロンプト）」と例示されていたが、Generator は以下の理由からローカル決定論的アルゴリズムを採用した:
  - 非弁行為リスク: LLM を通さないことで、どれだけ工夫しても「助言」が混入する余地がゼロ
  - API キー不要 / 完全オフライン / <1秒応答
  - 自動テスト可能（決定論的）
  - サンプルデータ 50 件規模では十分な精度が出る
- アルゴリズム:
  1. 平仮名・記号で文を区切って「カタカナ/漢字/英数の塊」を重要トークンとして抽出（長音符 ー は区切らない）
  2. ノイズトークン（事例・事案・架空・サンプル・納税者・課税庁など）を除外
  3. 3 文字以上の全漢字トークンは 2 文字 n-gram にも分解（「購入費」→購入/入費）
  4. 各事案を title(×5) / issue(×4) / relatedLaws(×3) / summary(×2) / fullText(×1) で重み付けスコア
  5. クエリに税目ヒント（フリーランス・相続・消費税など）が含まれていたら、その税目の全事案にシードスコア + 直接マッチにさらに加点
  6. スコア降順 → 日付降順 で上位 10 件
- **この設計は spec の「解釈・助言を出さない」「20秒以内」を原理的に保証する**。将来的に embedding 化や LLM ハイブリッド化する場合も、同じインターフェース（`naturalLanguageSearch(query, options)`）で差し替えられる形にしてある。

## 良かった点

- `?ask=` で URL 共有できるので、他人に「こう検索してみた」を送れる
- マッチ理由チップ（ヒット: フリーランス / 経費 + 税目ヒット）があり、なぜその事案が返ったか説明性が高い
- 自然文モード中は通常の税目・キーワードフォームを非表示にして迷わない UI
- LLM 非依存なので、API 障害・レート制限・API キー未設定でも常に動く
- token extractor・scoreCase・naturalLanguageSearch と責務分離されているので将来のハイブリッド化が容易

## 改善要望（MVP 後）

- 類義語辞典の拡充（源泉徴収⇔給与所得、減価償却⇔耐用年数 など）
- 実データ差し替え後は、サンプル 50 件では試せなかったスケール時の挙動を再評価
- 「似ている順」に加えて「日付順」「裁判所順」の二次ソート切替
- 現状はサーバーコンポーネントで即レンダリングだが、長いクエリでタイプラグが出るなら debounce 付きクライアントも検討
