## 3. 再開したらまず確認する「スモークテスト URL」

起動後、以下を順に開いて全部動いていれば MVP は健康。

| # | 何を見るか | URL | 合格ライン |
|---|-----------|-----|-----------|
| 1 | トップページが開く | `/` | 50 件表示、免責フッター、自然文検索フォーム、税目/キーワードフォームがすべて見える |
| 2 | 税目フィルタ（相続税）| `/?subcategory=inheritance_tax` | 相続税の事案のみ表示される |
| 3 | キーワード検索 | `/?q=必要経費` | 「必要経費」を含む事案が表示される |
| 4 | 空ヒット | `/?q=存在しない語` | 「該当する裁判例が見つかりませんでした」 |
| 5 | 詳細ページ | `/cases/sample-005` | ヘッダー情報・タブ・関連事案リンクが見える |
| 6 | 詳細タブ切替 | `/cases/sample-005?tab=full` | 全文タブが選択状態になる |
| 7 | 上訴チェーン | `/cases/sample-002` → 上訴先 `sample-005` → さらに上訴先 `sample-007` | リンクでたどれる |
| 8 | 404 | `/cases/does-not-exist` | カスタム 404 ページ + 「一覧に戻る」リンク |
| 9 | 自然文検索 A | `/?ask=フリーランスで経費にした車の購入費を否認された` | 所得税系が 9 件返る。先頭は sample-001 付近。マッチチップ＋「税目ヒット」バッジ |

| 11 | 空白のみ ask | `/?ask=%20%20%20` | 自然文モードに入らず通常 50 件表示 |
| 12 | モバイル幅 | DevTools で 375px 幅にして `/` を開く | 横スクロール無し |

上記のどれかが壊れていたら、`docs/feedback/sprint-N.md` を見直して該当スプリントの受け入れ基準に立ち返る。

## 4. 今、ユーザー判断待ちになっている選択肢

MVP は完成しているので、次は「MVP 後」の方向を決めるフェーズ。下記はいずれも
**CLAUDE.md ルール 6 により「着手前に確認が必要」** な種類のアクションなので、
自動では進めていない。再開時にどれを選ぶか決めてほしい。

1. **実データ投入**
   - 今は `src/data/cases.json` の架空サンプル 50 件。
   - 実判例・裁決例を入れる場合、データソース選定・ライセンス確認・スキーマ拡張が必要。
   - 「どこから何件を、どのフォーマットで入れるか」を決めるところから。

2. **仕様書（spec.md）の改訂 / Sprint 5 以降の企画**
   - 改善要望は `docs/feedback/sprint-4.md` 末尾に列挙済み:
     - 類義語辞典の拡充
     - ソート切替（日付順／裁判所順）
     - 入力のデバウンス化
     - embedding / LLM ハイブリッド化
   - どれを次スプリントに入れるかは Planner に追記依頼が必要。

3. **デプロイ**
   - 現在はローカル `npm run dev` 動作のみ。
   - Vercel 等に出す場合は本番 `next build` が通るかの確認と、ドメイン・環境変数の決定。

4. **ユーザーヒアリング**
   - 実際の対象ユーザー（税務で困った個人／税理士の卵 etc.）に触ってもらうフェーズ。
   - コード作業ではなく運用・調査系。

5. **非機能面のハードニング**
   - 入力 XSS・長文 ask の DoS 耐性・検索ログ保存有無などを詰める。

**再開時にやること: 上記 1〜5 のどれを優先するかを指示する。** それまでコードには手を入れない。

## 6. 「また壊れてないか」だけ確認したい時の最短手順

1. `start.command` をダブルクリック
2. ブラウザで `/?ask=フリーランスで経費にした車の購入費を否認された` を開く
3. 所得税系の事案が複数返ってきて、各行に青い「ヒット」チップと緑の「税目ヒット」バッジが見えていれば、
   **Sprint 1〜4 の主要経路はすべて健康**（このクエリは全スプリントの実装を横断的に使う）。
4. 先頭事案の行をクリック → 詳細ページが開き、タブ切替・関連事案リンクが動けば OK。
